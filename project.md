## 1. プロジェクト概要

### 1.1 目的

TickTickにインスパイアされた高機能タスク管理 PWA を個人向けに開発する。単なるTODOリストに留まらず、「セカンドブレイン」の思想を取り入れることで、タスク・プロジェクト・メモ・リソースを一元管理する「思考の外部化システム」として設計する。

### 1.2 ターゲットユーザー

- PKM（Personal Knowledge Management）に興味を持つ個人ユーザー
- TickTick / Todoist / Notion などのツールを使い慣れたユーザー
- 日本語環境をメインで使用するユーザー

### 1.3 コアバリュー

| 価値 | 説明 |
|------|------|
| **Capture（捕捉）** | どこでもすばやくアイデア・タスクを記録できる |
| **Organize（整理）** | PARAメソッドで情報を構造化する |
| **Distill（蒸留）** | 重要な情報を前面に出し、ノイズを下げる |
| **Express（発信）** | マークダウンやページ形式で思考をアウトプットする |

## 2. 設計思想

### 2.1 セカンドブレイン（Second Brain）

Tiago Forte の「Building a Second Brain」に基づく設計。脳の外側に信頼できる知識管理システムを構築することで、認知負荷を下げ、クリエイティブな思考に集中できる環境を実現する。

### 2.2 CODEメソッド

アプリのコアワークフローは以下の4ステップで設計する：

```
Capture（捕捉）→ Organize（整理）→ Distill（蒸留）→ Express（発信）
```

| ステップ | アプリでの実装 |
|----------|---------------|
| **Capture** | インボックス（クイックキャプチャ）、スラッシュコマンド、ハッシュタグ |
| **Organize** | PARAフォルダ構造、タグ、フィルター |
| **Distill** | ピン留め、重要度フラグ、フォーカスモード |
| **Express** | マークダウンエディタ、ページ形式のタスク詳細 |

### 2.3 PARAメソッド

情報を以下の4カテゴリで分類・管理する：

| カテゴリ | 説明 | 例 |
|----------|------|-----|
| **Projects（プロジェクト）** | 期限のある目標。完了したら終わる | 「サイトリニューアル」「確定申告」 |
| **Areas（エリア）** | 継続的に維持する責任領域 | 「健康管理」「家計」「学習」 |
| **Resources（リソース）** | 将来役立つかもしれない参考情報 | 記事、ブックマーク、メモ |
| **Archives（アーカイブ）** | 完了・非アクティブな項目の保管庫 | 完了プロジェクト、過去メモ |

## 4. UI・UX設計

### 4.1 レイアウト方針

- **3ペイン構成**：サイドバー（ナビ）+ コンテンツリスト + 詳細ペイン
- サイドバーはコラプス可能（モバイル時は非表示 → ハンバーガーメニュー）
- 詳細ペインはフルスクリーンモードに展開可能（Notionライク）
- カラーテーマ：ライト / ダーク / システム追従

### 4.2 ビュー切り替え

| ビュー | 説明 |
|--------|------|
| **リストビュー** | デフォルト。TickTick風の縦型タスクリスト |
| **ボードビュー** | カンバン形式（ステータス列） |
| **カレンダービュー** | 実行日・締め切り日をカレンダー上に表示 |
| **タイムラインビュー** | プロジェクト全体の工程を横断的に確認（将来フェーズ） |

### 4.3 クイックキャプチャ

- グローバルショートカット（`Cmd/Ctrl + K` など）でどの画面からでもキャプチャモーダルを起動
- モーダルで入力 → Enterで即座にインボックスへ保存
- キャプチャ時にプロジェクト・タグ・日付を付与可能

### 4.4 TickTick踏襲UIポイント

- タスクの完了チェックボックスをクリックするとアニメーションで消える
- 優先度は4段階（緊急 / 高 / 中 / なし）、色で視覚的に識別
- サブタスク（チェックリスト）の入れ子対応
- 右クリックコンテキストメニューによる操作
- ドラッグ＆ドロップによるタスク順序変更・プロジェクト間移動

## 5. データモデル

### 5.1 Task（タスク）

```typescript
interface Task {
  id: string;                    // UUID
  title: string;                 // タスク名（ページタイトル）
  body: string;                  // マークダウン本文（詳細メモ）
  
  // 日付管理（要件の核心）
  dueDate: Date | null;          // 締め切り日
  scheduledDate: Date | null;    // 実行日（いつやるか）
  completedAt: Date | null;      // 完了日時
  
  // 分類
  projectId: string | null;      // 所属プロジェクト
  areaId: string | null;         // 所属エリア
  tags: string[];                // タグ一覧
  
  // 状態
  status: 'inbox' | 'todo' | 'in_progress' | 'done' | 'archived';
  priority: 0 | 1 | 2 | 3;      // 0=なし, 1=中, 2=高, 3=緊急
  
  // サブタスク
  parentId: string | null;
  subtasks: Subtask[];
  
  // メタデータ
  createdAt: Date;
  updatedAt: Date;
  userId: string;
  
  // 外部連携（オプション）
  externalSource?: {
    provider: 'notion' | 'craft';
    externalId: string;
    url?: string;
  };
}

interface Subtask {
  id: string;
  title: string;
  completed: boolean;
  order: number;
}
```

## 6. 機能要件

### 6.1 タスク管理

#### 6.1.1 基本操作

| 機能 | 詳細 |
|------|------|
| タスク作成 | クイックキャプチャ、リスト末尾のインライン入力、詳細ペイン |
| タスク編集 | タイトル・本文・全プロパティのインライン編集 |
| タスク完了 | チェックボックスで完了トグル、完了済みは折り畳み表示 |
| タスク削除 | ゴミ箱へ移動（30日後に完全削除） |
| タスク複製 | コンテキストメニュー or ショートカット |
| タスク移動 | D&Dまたはコンテキストメニューでプロジェクト間移動 |

#### 6.1.2 日付管理（重要）

- **締め切り日（Due Date）**：タスクの期限。期限超過で赤色表示。
- **実行日（Scheduled Date）**：実際に作業を実行する予定日。「今日」ビューに表示される基準日。
- 両者は独立したプロパティ。「実行日=明日、締め切り=来週」のような設定が可能。
- カレンダービューでは、実行日でタスクを配置し、締め切り日は別アイコンで表示。

#### 6.1.3 繰り返しタスク

- 毎日 / 毎週（曜日指定） / 毎月（日付指定） / 毎年
- カスタム繰り返し（例：3日ごと）
- 完了時に次のタスクを自動生成

#### 6.1.4 サブタスク・チェックリスト

- タスク詳細内でサブタスクをインライン追加
- サブタスクのD&Dによる並び替え
- サブタスクの完了率をリストビューに進捗バーで表示

### 6.2 プロジェクト・エリア管理

- PARAカテゴリへの割り当て
- プロジェクトのステータス管理（進行中 / 完了 / アーカイブ）
- プロジェクトをアーカイブすると自動的にArchivesカテゴリへ
- エリアは期限なし・常時アクティブな領域として管理

### 6.3 フィルタリング・検索

| 機能 | 詳細 |
|------|------|
| グローバル検索 | タイトル・本文・タグ全文検索、`Cmd/Ctrl+K` |
| フィルター | 優先度・タグ・担当・ステータス・日付範囲 |
| スマートリスト | 「今日」「近日」「期限切れ」「未分類（インボックス）」 |
| ソート | 手動 / 期限日 / 作成日 / 優先度 |

### 6.4 フォーカスモード（Distill）

- 選択したタスクのみをフルスクリーンで表示
- ポモドーロタイマー内蔵（25分作業 / 5分休憩）
- フォーカス中はサイドバー・通知を非表示

## 7. エディタ・入力機能

### 7.1 マークダウンエディタ

- WYSIWYGモード（プレビュー）と Rawマークダウンモードのトグル
- 対応要素：見出し（H1〜H3）、太字、斜体、コード、インラインコード、引用、リスト（番号付き/なし）、チェックボックスリスト、テーブル、水平線
- シンタックスハイライト（コードブロック）

### 7.2 キーボードショートカット

#### グローバル

| ショートカット | 動作 |
|--------------|------|
| `Cmd/Ctrl + K` | クイックキャプチャ / グローバル検索 |
| `Cmd/Ctrl + N` | 新規タスク作成 |
| `Cmd/Ctrl + /` | ショートカット一覧を表示 |

#### タスクリスト

| ショートカット | 動作 |
|--------------|------|
| `Enter` | タスク詳細を開く / 次のタスクを追加 |
| `Space` | 選択タスクを完了トグル |
| `Delete / Backspace` | タスクを削除（確認あり） |
| `↑ / ↓` | タスク選択移動 |
| `Cmd/Ctrl + ↑/↓` | タスクを上下に並び替え |
| `Tab` | サブタスクにインデント |
| `Shift + Tab` | インデント解除 |

#### エディタ内

| ショートカット | 動作 |
|--------------|------|
| `Cmd/Ctrl + B` | 太字 |
| `Cmd/Ctrl + I` | 斜体 |
| `Cmd/Ctrl + E` | インラインコード |
| `Cmd/Ctrl + Shift + K` | コードブロック |
| `Cmd/Ctrl + Shift + 7` | 番号付きリスト |
| `Cmd/Ctrl + Shift + 8` | 箇条書きリスト |
| `Cmd/Ctrl + Shift + 9` | チェックボックスリスト |

### 7.3 スラッシュコマンド（/ コマンド）

エディタ内で `/` または `;`（日本語環境向け）を入力するとコマンドパレットが起動する。

| コマンド | 動作 |
|---------|------|
| `/今日` / `/today` | 実行日を今日に設定 |
| `/明日` / `/tomorrow` | 実行日を明日に設定 |
| `/期限` / `/due` | 締め切り日ピッカーを開く |
| `/優先度` / `/priority` | 優先度セレクターを開く |
| `/タグ` / `/tag` | タグ入力 |
| `/プロジェクト` / `/project` | プロジェクト選択 |
| `/見出し1` / `/h1` | H1見出しブロック挿入 |
| `/見出し2` / `/h2` | H2見出しブロック挿入 |
| `/コード` / `/code` | コードブロック挿入 |
| `/テーブル` / `/table` | テーブル挿入 |
| `/区切り` / `/divider` | 水平線挿入 |
| `/サブタスク` / `/sub` | サブタスクチェックリスト挿入 |

**実装注意**：IME（日本語入力）確定前の `;` を検知してコマンド起動。`compositionend` イベント後に `;` であればコマンドパレット起動とする。

### 7.4 ハッシュタグ（#）タグ管理

- タスク入力時に `#タグ名` で即座にタグ付け
- 既存タグはサジェスト表示
- 存在しないタグは新規作成
- サイドバーのタグ一覧から一覧・クリックでフィルタリング

### 7.5 アットメンション（@）日付メンション

Notionライクな日付メンション機能。

- `@今日` → 今日の日付をインライン表示
- `@明日` → 明日の日付
- `@YYYY-MM-DD` → 任意の日付
- `@` 入力後に日付ピッカーが起動し、選択するとテキストに埋め込まれる
- 日付メンションをクリックするとカレンダービューへジャンプ

### 7.6 ページ構造（Notionライク）

- タスクは「ページ」として扱い、タイトルがタスク名になる
- タイトル下の本文エリアに自由にマークダウンで記述可能
- サブページ（ネストされたタスク）を作成し、階層構造を構築可能
- ページ上部にプロパティパネル（締め切り日・実行日・タグ等）を表示

## 8. 外部連携

### 8.1 連携方針

```
[アプリ単体モード] ← デフォルト
  すべてのデータをアプリ内DBに保存

[ハイブリッドモード] ← 外部連携有効時
  外部サービスのデータはAPIで都度取得（キャッシュあり）
  アプリ内データはアプリ内DBに保存
  両者を統合して表示
```

### 8.2 カレンダー連携

| サービス | 連携方法 | 読み書き |
|---------|---------|---------|
| **Googleカレンダー** | Google Calendar API v3 + OAuth2 | 読み取り（将来は書き込みも） |
| **Outlookカレンダー** | Microsoft Graph API + OAuth2 | 読み取り |
| **iCloudカレンダー** | CalDAV プロトコル | 読み取り |

- 設定画面に各サービスへの「連携する」ボタンを設置
- 連携後はカレンダービューにイベントをオーバーレイ表示
- イベントのキャッシュ有効期間：30分（設定変更可）
- プライバシー：アクセストークンはユーザーデバイス or サーバーの暗号化ストレージに保管

### 8.3 タスク管理連携

#### Notion

- Notion API（公式）を使用
- 設定：APIキー + 対象データベースIDを入力
- 機能：指定データベースのページ一覧を取得 → タスクとして表示
- 取得項目：タイトル・ステータス・期日・タグ
- 書き込み：アプリで作成したタスクをNotionに同期（オプション）

#### Craft

- Craft公式APIを使用
- 設定：APIキー + スペースIDを入力
- 機能：Craftのドキュメント一覧を取得 → リソースとして表示
- 主にResourcesカテゴリとの連携を想定

### 8.4 メモ連携

Apple純正メモ・Google KeepはAPIが非公開のため対象外とし、OSS代替を採用する。

#### Memos（OSS）

- セルフホスト型メモアプリ（grpc/REST API対応）
- 設定：エンドポイントURL + APIトークン
- 機能：メモ一覧取得 → Resourcesカテゴリに表示
- 双方向同期（メモの作成・更新もMemos側に反映）

#### Obsidian

- Obsidian Local REST API プラグインを経由
- 設定：ローカルAPIエンドポイント + APIキー
- 機能：Vault内のマークダウンファイル一覧を取得
- 主にResourcesカテゴリへのリンクとして表示
- **注意**：Obsidianはローカル起動が必要なため、PC使用時のみ有効

## 9. 認証・同期

### 9.1 認証

- **対応プロバイダー**：Googleログインのみ（OAuth2 / OIDC）
- メールアドレス・パスワード認証は提供しない
- セッション管理：JWT（アクセストークン短命 + リフレッシュトークン長命）

### 9.2 データ同期

#### アプリ内データ（ローカル ↔ クラウド）

```
[ブラウザ / PWA]
  IndexedDB（オフラインキャッシュ） ←→ [同期エンジン] ←→ [クラウドDB]
```

- **オフライン対応**：ネットワーク切断中は IndexedDB に書き込み
- **再接続時**：差分を自動同期（Conflict Resolution：後勝ち + タイムスタンプ比較）
- **マルチデバイス**：WebSocket or Server-Sent Events でリアルタイム反映

#### 外部連携データ（読み取り系）

- 外部サービスのデータはアプリ内DBに永続保存しない

- キャッシュ失効後は再取得

### §15-A  Craft API — タスク機能の連携可否調査

**調査日**: 2026-05-30  
**ステータス**: 部分確認済み・要追加調査

#### 確認できた事実

Craft公式ドキュメントおよびヘルプサイトより以下を確認：

- **公式の記載**："Daily notes and tasks are fully accessible via API"（タスクはAPIから完全にアクセス可能と明記）
- **API認証方式**：スペースごとにCraft設定画面でAPI接続を有効化 → ユニークなAPIエンドポイントと認証情報が発行される
- **APIの形式**：REST（エンドポイントはスペースごとに固有のURLが割り当てられる形式）

  - ドキュメントのCRUD（作成・更新・削除）
  - コレクション管理
  - **Daily Notesとタスクへのアクセス**（詳細エンドポイント仕様は未確認）

#### タスク固有の仕様（未確認項目）

公開APIドキュメント（`connect.craft.do/api-docs`）では「Daily Notes and Tasks」として専用カテゴリの存在が確認できるが、具体的な以下の仕様はドキュメントが認証壁の内側にあり確認できていない：

- タスクオブジェクトのJSONスキーマ（フィールド一覧）

- タスクのみを抽出するクエリパラメータ

### §15-B  iCloudカレンダー（CalDAV）のCORS問題と実装方法

**調査日**: 2026-05-30  
**ステータス**: 実装方針確定

#### 問題の概要

ブラウザから直接 `caldav.icloud.com` にリクエストを送ると、iCloud側がCORSヘッダーを返さないため **CORS エラーでブロック**される。PWAはブラウザ上で動作するため、この問題は回避できない。

#### 解決策：サーバーサイドプロキシ

バックエンドサーバー（Next.js API Route）がCalDAVリクエストを中継する。

```
[ブラウザ/PWA]
    ↕ REST (HTTPS, CORS OK)
[Next.js API Route /api/calendar/icloud]
    ↕ CalDAV/PROPFIND (サーバー間通信, CORSなし)
[caldav.icloud.com]
```

#### 実装技術

| 要素 | 採用候補 |
|------|---------|
| CalDAVクライアント | **tsdav**（Node.js用CalDAV/WebDAVクライアント、iCloud対応済み） |
| 認証 | iCloud App-Specific Password（アプリ固有パスワード） ＋ Apple ID |
| プロキシ実装 | Next.js API Route（サーバーレス関数） |
| トークン保管 | Supabase の暗号化カラム（pgcrypto）|

#### 難易度評価

| 項目 | 評価 |
|------|------|
| 技術的複雑さ | ★★★☆☆（中程度）|
| ライブラリの成熟度 | ★★★★☆（tsdavは実績あり） |
| iCloud認証の煩雑さ | ★★★★☆（App-Specific Passwordが必要） |
| ユーザーへの説明コスト | ★★★☆☆（設定手順が独特） |

**結論**: 実装可能。tsdavを使ったサーバーサイドプロキシ方式で技術的な障壁はない。ただしユーザーにはApple IDのApp-Specific Passwordを発行してもらう必要があり、設定UIで手順をガイドする必要がある。

#### ユーザーへの設定フロー

1. appleid.apple.com でアプリ固有パスワードを発行
2. TaskMapの設定画面でApple ID + アプリ固有パスワードを入力
3. バックエンドが暗号化して保管し、以降はプロキシ経由でiCloudカレンダーを取得

### §15-C  Obsidian Local REST API — 連携仕様

**調査日**: 2026-05-30  
**ステータス**: 仕様確認済み・制約事項明確化

#### プラグイン概要

[obsidian-local-rest-api](https://github.com/coddingtonbear/obsidian-local-rest-api) はObsidianのコミュニティプラグイン。Vault内のノートをREST API経由で操作できる。

#### 主なAPI機能

| 機能 | エンドポイント | 詳細 |
|------|-------------|------|
| ノート一覧取得 | `GET /vault/` | Vault内のファイル一覧 |
| ノート読み取り | `GET /vault/{path}` | 指定ファイルの内容取得 |
| ノート作成・更新 | `PUT /vault/{path}` | ファイルの作成・上書き |
| ノート削除 | `DELETE /vault/{path}` | ファイル削除 |
| セクション部分更新 | `PATCH /vault/{path}` | 見出し・ブロック単位の差分更新 |
| 全文検索 | `POST /search/simple/` | キーワード検索 |
| メタデータ検索 | `POST /search/` | JsonLogicによるフロントマター検索 |

#### 認証

- HTTPS（自己署名証明書）+ APIキー認証
- プラグイン設定画面でAPIキーを生成し、リクエストヘッダーに付与

#### 制約事項（重要）

| 制約 | 内容 |
|------|------|
| **デスクトップ限定** | ObsidianのデスクトップアプリがローカルPC上で起動している必要がある |
| **モバイル不可** | iOS/Android版Obsidianはローカルサーバーを立てられないため対象外 |
| **ローカルネットワーク** | デフォルトは `localhost:27123`。外部からアクセスするにはポートフォワードかngrokが必要 |
| **証明書警告** | 自己署名証明書のため、ブラウザから直接呼ぶと証明書エラーが発生する可能性あり |

#### TaskMapでの実装方針

- **接続方式**：ブラウザから直接呼ぶのではなく、Next.js API Routeがプロキシとして中継
- **ユーザー設定**：設定画面でObsidianのAPIエンドポイント（デフォルト: `http://localhost:27123`）とAPIキーを入力
- **用途**：ResourcesカテゴリへのVaultノートの取り込み（読み取りメイン）
- **制限の明示**：設定画面に「PCでObsidianが起動中の場合のみ機能します」と案内を表示
- **Obsidian Sync利用者**：Obsidian Sync（有料）でVaultを複数デバイスに同期している場合、デスクトップPC起動中であればモバイルのTaskMapからも間接的に参照可能

### §15-D  データ同期コンフリクト解決とチームプライバシー設計

**調査日**: 2026-05-30  
**ステータス**: 方針確定

#### コンフリクト解決戦略

##### 個人利用フェーズ（Phase 1〜3）

**後勝ち（Last Write Wins）方式** で十分と判断する。

| 理由 | 説明 |
|------|------|
| 単一ユーザー | 自分のデータを自分だけが編集するため、真の競合は起きにくい |
| 非同期利用パターン | オフラインで編集 → オンラインで同期、という利用が多く競合頻度が低い |
| 実装コスト | OT/CRDTは実装複雑度が高く、個人利用では費用対効果が低い |

**実装**：各レコードに `updatedAt` タイムスタンプを付与し、新しい方を正とする。

##### チーム機能フェーズ（Phase 4）

複数人が同じタスクを同時編集する可能性が生じるため、**CRDT（Conflict-free Replicated Data Type）** の採用を検討する。

| 選択肢 | 特徴 |
|--------|------|
| **Yjs** | 最も実績あるCRDTライブラリ。Tiptapと直接統合可能 |
| **Automerge** | JSONドキュメント全体をCRDT管理。より汎用的 |
| **Supabase Realtime** | DBレベルのリアルタイム同期（行単位）。CRDTではないがシンプル |

**推奨**：フェーズ4ではまずSupabase Realtimeで実装し、同時編集の競合が実問題になった段階でYjsを導入する段階的アプローチを採用する。

#### チーム共有時のプライバシー設計

##### スペース分離モデル

```
ユーザーA
├── 個人スペース（プライベート）   ← チームメンバーは一切見えない
│   ├── Projects / Areas / Resources
│   └── 外部連携データ（Notion, Craft等）
└── チームスペース（共有）         ← ロールに応じてアクセス制御
    ├── 共有プロジェクト
    └── 共有タスク
```

##### 外部連携とチームの関係

| 状況 | 動作 |
|------|------|
| Notionを個人連携している場合 | Notionデータは個人スペースにのみ表示される |
| チームメンバーがいる場合 | メンバーはNotionデータを閲覧できない（オーナーの認証情報を共有しない） |
| NotionデータをチームにSharedしたい場合 | 「チームスペースにインポート」という明示的操作でTaskMap内にコピーを作成する。以降はCraftやNotionではなくTaskMapのDBが正となる |
| Craftも同様 | Craft連携はオーナー個人の設定。チームへの自動公開なし |

**結論**：外部連携データは個人のOAuth認証に紐づくため、チームメンバーへの漏洩リスクは構造上ない。ただしTaskMapにインポートしたデータはTaskMapのアクセス制御に従う点を明示する。

##### DBレベルの実装

- **Supabase Row Level Security（RLS）** を全テーブルに適用
- 個人スペースのレコード：`userId = auth.uid()` のみ許可
- チームスペースのレコード：`workspaceId IN (ユーザーが所属するワークスペース一覧)` でフィルタリング
- ロール（オーナー/編集者/閲覧者）は `workspace_members` テーブルで管理
